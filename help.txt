 Что каждый трейдер может настроить под себя

  Все настройки именные, привязаны к его client_id. Три параметра, безопасных без переобучения:

  ┌─────────────────────┬──────────┬─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │      Параметр       │ Диапазон │                                                               Что делает                                                                │
  ├─────────────────────┼──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ signal_threshold    │ 0.0 —    │ Порог уверенности для входа в LONG/SHORT. По умолчанию 0.40. Ниже = больше сигналов / больше шума. Выше = редкие, но более уверенные.   │
  │                     │ 1.0      │                                                                                                                                         │
  ├─────────────────────┼──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ max_signal_time_sec │ 1 — 3600 │ Максимальное ожидаемое время движения, при котором сигнал считается актуальным. По умолчанию 600. Скальпер ставит 60–120, свинг ставит  │
  │                     │          │ 600+.                                                                                                                                   │
  ├─────────────────────┼──────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ sample_every_ticks  │ 1 —      │ Как часто модель делает предсказание (каждый N-й тик). По умолчанию 50. Меньше = чаще считать, нагрузка выше.                           │
  │                     │ 10000    │                                                                                                                                         │
  └─────────────────────┴──────────┴─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

 1) Посмотри, что видит детектор твоего клиента

  curl "http://127.0.0.1:5009/state?client_id=trader-1"

  Ключевое поле — direction:
  - -1 — детектор думает «мы в нисходящем тренде» → модели на вход подаётся direction=-1, она будет тянуть в SHORT.
  - +1 — детектор считает что мы в лонге. 
  - 0 — ещё не определился (редко).
  
  Если рынок реально лонгует 30 минут, а direction=-1 — значит детектор «застрял» от старого seed-состояния и ещё не увидел
  отката нужного размера. Что делать:

  2) Сбрось состояние и пусть детектор поймает текущий тренд по живому потоку

  curl -X POST "http://127.0.0.1:5009/reset?client_id=trader-1" \
    -H "Content-Type: application/json" \
    -d '{"use_seed": false}'

  После этого direction начнёт с 0, и как только цена пройдёт 0.30 % в одну сторону — встанет в актуальное направление. Минус:
  придётся снова накопить 128 тиков для предсказаний (3–5 минут на BTCUSDT).

  3) Если рынок «вроде лонг», но не очевидно

  Подними порог, чтобы лишний раз не дергало, и заодно отсей разворотные сигналы:

  curl -X POST http://127.0.0.1:5009/config \
    -H "Content-Type: application/json" \
    -d '{"client_id": "trader-1", "signal_threshold": 0.55, "max_signal_time_sec": 300}'


  Таким образом получаешь именно «сколько трейдеров — столько вариантов»: модель одна и та же (веса, признаки, окно —
  фиксированы), а решающее правило, частота сэмплинга и допустимый горизонт у каждого свои.



 Текущая цена с биржи Bybit (BTCUSDT, фьючерс)

  Запросить тикер по бессрочному фьючерсу BTCUSDT (category=linear — линейные перпетуалы USDT):
  curl "https://api.bybit.com/v5/market/tickers?category=linear&symbol=BTCUSDT"

  Ответ (сокращённо):
  {
    "retCode": 0,
    "retMsg": "OK",
    "result": {
      "category": "linear",
      "list": [
        {
          "symbol": "BTCUSDT",
          "lastPrice": "65700.00",
          "bid1Price": "65699.90",
          "ask1Price": "65700.10",
          "markPrice": "65700.05"
        }
      ]
    }
  }

  Для ПОКУПКИ важна цена ask (ask1Price) — именно по ней исполнится маркет-buy.
  Для продажи — bid (bid1Price). lastPrice — цена последней сделки.

  Вытащить только цену покупки (нужен jq):
  curl -s "https://api.bybit.com/v5/market/tickers?category=linear&symbol=BTCUSDT" \
    | jq -r '.result.list[0].ask1Price'



 Запрос предсказания у сервера модели (endpoint /tick)

  Шлёшь очередной тик рынка — в ответ получаешь предсказание модели для своего client_id.
  Один тик = одна сделка с биржи (timestamp, side, size, price):

  curl -X POST http://127.0.0.1:5009/tick \
    -H "Content-Type: application/json" \
    -d '{
      "client_id": "trader-1",
      "timestamp": 1781514080.5,
      "side": "Buy",
      "size": 0.123,
      "price": 65700.0
    }'

  Ответ сервера (реальная схема):

  - Пока модель не готова или нет сигнала на этом тике:
    {"message": "Сигнал ещё не готов"}

  - Когда модель выдаёт сигнал:
    {
      "signal": "SHORT",                 # направление: "LONG" / "SHORT"
      "predicted_time_sec": 427.49,      # ожидаемое время движения / актуальность сигнала, сек
      "prob_short": 0.6399,              # вероятность SHORT (softmax)
      "prob_hold":  0.2975,              # вероятность HOLD (вне рынка)
      "prob_long":  0.0626,              # вероятность LONG
      "price": 62634.1,                  # цена тика, на котором сделано предсказание
      "timestamp": 1781844609.652        # время тика, сек
    }

  У модели две головы: классификация направления (prob_short/prob_hold/prob_long, из них
  выбирается signal) и регрессия времени (predicted_time_sec).

  Как читать:
  - signal             — направление входа: "LONG" (покупка по ask) / "SHORT" (продажа по bid).
  - predicted_time_sec — СКОЛЬКО ВРЕМЕНИ сигнал актуален, в секундах (вторая голова; ~250 сек).
  - prob_short/hold/long — вероятности по классам, приходят С СЕРВЕРА.
  - message            — есть, когда сигнала нет: не накопилось lookback_ticks (128) тиков, либо
                         тик не попал в точку сэмплинга (sample_every_ticks), либо уверенность
                         ниже signal_threshold.

  ВНИМАНИЕ: «уверенность» (conf) — это НЕ поле сервера. Сервер присылает только prob_short/
  prob_hold/prob_long. «Уверенность» мы считаем сами в скрипте как max из этих трёх вероятностей
  (= вероятность того направления, что стало сигналом). Именно это max сервер сравнивает с
  signal_threshold: если max >= порога — отдаёт signal, иначе "Сигнал ещё не готов".

  Поэтому в живом потоке нормально видеть много "сигнала нет" и изредка сигналы.
  Разобрать ответ человеческими словами (направление, уверенность, время):
    echo "$RESPONSE" | jq -rc '
      if .signal then
        (if .signal=="SHORT" then "ШОРТ (продажа)" elif .signal=="LONG" then "ЛОНГ (покупка)" else .signal end) as $dir
        | "\($dir)  уверенность \(([.prob_short,.prob_hold,.prob_long]|max)*100|round)%  актуально ~\(.predicted_time_sec|round) сек"
      else .message end'

  Связать с биржей: цену price для тика берёшь из Bybit (lastPrice/ask1Price выше), шлёшь в /tick,
  и если пришёл signal="LONG" — покупаешь по ask1Price.



 Скрипт .sh: цена с Bybit -> сервер предсказаний -> ответ модели

  Сохрани как predict_once.sh, сделай chmod +x predict_once.sh, запусти ./predict_once.sh
  Нужны установленные curl и jq.

  #!/usr/bin/env bash
  set -euo pipefail

  CLIENT_ID="trader-1"
  PREDICT_URL="http://127.0.0.1:5009/tick"
  BYBIT_URL="https://api.bybit.com/v5/market/tickers?category=linear&symbol=BTCUSDT"

  # 1) Берём текущую цену покупки (ask) с Bybit
  PRICE=$(curl -s "$BYBIT_URL" | jq -r '.result.list[0].ask1Price')
  if [ -z "$PRICE" ] || [ "$PRICE" = "null" ]; then
    echo "Не удалось получить цену с Bybit" >&2
    exit 1
  fi
  echo "Цена покупки (ask) с Bybit: $PRICE"

  # 2) Отправляем цену на сервер предсказаний как тик
  TS=$(date +%s)
  RESPONSE=$(curl -s -X POST "$PREDICT_URL" \
    -H "Content-Type: application/json" \
    -d "{
      \"client_id\": \"$CLIENT_ID\",
      \"timestamp\": $TS,
      \"side\": \"Buy\",
      \"size\": 0.001,
      \"price\": $PRICE
    }")

  # 3) Печатаем ответ модели
  echo "Ответ модели:"
  echo "$RESPONSE" | jq .

  # Коротко: сигнал, иначе сообщение «не готов»
  echo "$RESPONSE" | jq -r '.signal // .message'

  ВАЖНО: один запуск шлёт один тик. Модели нужно накопить lookback_ticks (128) тиков,
  поэтому на первом тике сервер отвечает {"message":"Сигнал ещё не готов"}. Чтобы довести
  модель до готовности, нужен ЖИВОЙ ПОТОК сделок (см. WebSocket-скрипт ниже), а не повтор
  одной и той же REST-цены — на одинаковых ценах предсказание бессмысленно.



 Скрипт .sh: ЖИВОЙ поток сделок Bybit (WebSocket) -> сервер предсказаний

  Это правильный способ «накормить» модель: подписываемся на поток реальных сделок
  publicTrade.BTCUSDT и каждую сделку отправляем в /tick. Через ~128 сделок появится
  ready=true и начнут приходить сигналы.

  Нужны: websocat (brew install websocat), jq, curl.
  Сохрани как stream_ticks.sh, chmod +x stream_ticks.sh, запусти ./stream_ticks.sh
  Остановка — Ctrl+C.

  #!/usr/bin/env bash
  set -euo pipefail

  CLIENT_ID="trader-1"
  PREDICT_URL="http://127.0.0.1:5009/tick"
  WS_URL="wss://stream.bybit.com/v5/public/linear"
  SUB='{"op":"subscribe","args":["publicTrade.BTCUSDT"]}'

  # echo "$SUB" — подписка; cat — держит stdin открытым, чтобы соединение не закрылось.
  { echo "$SUB"; cat; } | websocat -n "$WS_URL" | while IFS= read -r MSG; do
    # одно сообщение publicTrade может содержать несколько сделок в .data[]
    echo "$MSG" | jq -c '.data[]? // empty' | while IFS= read -r TRADE; do
      PRICE=$(echo "$TRADE" | jq -r '.p')         # цена сделки
      SIDE=$(echo  "$TRADE" | jq -r '.S')         # Buy / Sell (агрессор)
      SIZE=$(echo  "$TRADE" | jq -r '.v')         # объём
      TS=$(echo    "$TRADE" | jq -r '.T/1000')    # время сделки, мс -> сек

      RESP=$(curl -s -X POST "$PREDICT_URL" \
        -H "Content-Type: application/json" \
        -d "{\"client_id\":\"$CLIENT_ID\",\"timestamp\":$TS,\"side\":\"$SIDE\",\"size\":$SIZE,\"price\":$PRICE}")

      # при сигнале — направление, уверенность (max из prob_*, считаем тут) и время; иначе «нет».
      # Все поля защищены от null, чтобы одна нестандартная сделка не уронила поток.
      echo "$RESP" | jq -rc --arg p "$PRICE" '
        if .signal then
          (if .signal=="SHORT" then "ШОРТ (продажа)" elif .signal=="LONG" then "ЛОНГ (покупка)" else "\(.signal)" end) as $dir
          | ([.prob_short, .prob_hold, .prob_long] | map(. // 0) | max) as $conf
          | (.predicted_time_sec // null) as $t
          | "цена=\($p)  \($dir)  уверенность \(($conf*100)|round)%  актуально "
            + (if ($t|type)=="number" then "~\($t|round) сек (\((($t/60)*10)|round/10) мин)" else "время неизвестно" end)
        else
          "цена=\($p)  сигнала нет (\(.message // "пустой ответ"))"
        end' || echo "цена=$PRICE  (не разобран ответ сервера)"
    done
  done

  Что увидишь: много строк "цена=... сигнала нет" и изредка сигнал, например:
  цена=62634.10  ШОРТ (продажа)  уверенность 64%  актуально ~427 сек (7.1 мин)
  -> шорт, уверенность 64%, ожидаемое время движения ~427 сек (~7 мин).
  Проверить состояние клиента можно параллельно:
  curl -s "http://127.0.0.1:5009/state?client_id=trader-1" | jq .
